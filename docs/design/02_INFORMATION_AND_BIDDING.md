# 02. Information and Bidding Design

> 文档职责：解释信息系统与竞价系统如何互相咬合。  
> 具体规则实现以 `01_MVP_RULE_SPEC.md` 为准。

---

## 1. Design Problem

竞价游戏最容易出现两个问题：

1. **信息不足**：玩家不知道该怎么估价，只能乱猜。
2. **承诺不足**：玩家可以随便报价、试探、撤回，导致前几轮失去意义。

《Bid or Bust》的设计目标是让每一轮都同时存在：

- 新信息。
- 新价格信号。
- 直接成交压力。
- 无法撤回的承诺风险。

---

## 2. Information Layers

当前设计有三层有效信息。

### 2.1 Public Intel / 公共信息

所有玩家同时看到。

来源：

- 每轮系统自动公开。
- 到期的技能信息。
- 成交后的总价值与净收益。

作用：

- 给所有玩家共同推理基础。
- 保证局面随轮次推进。
- 防止完全依赖技能。

### 2.2 Private Look / 私人查看

每名仍在局内的玩家每轮可查看 1 件物品。

作用：

- 让每个玩家都有不同样本。
- 即使没有技能也能产生信息差。
- 形成“我看到的”和“别人行为”的冲突。

### 2.3 Active Skill Intel / 主动技能信息

玩家主动选择是否使用技能，但不能选择具体触发哪个技能。

作用：

- 产生一轮信息优势。
- 使用技能本身成为桌面信号。
- 随机触发避免固定最优解。
- 下一轮公开避免长期信息压制。

## 3. Structured Information Principle

所有会影响竞价的信息必须是结构化信息。

### 3.1 Good Information

好的信息可被程序理解，并能用于候选过滤或保底价值计算。

示例：

```text
显示随机 2 件物品的品质。
显示最高品质物品的品质。
显示某件物品的分类。
显示箱中是否存在 Oddity。
显示最大物品是否为 Collectible。
显示一个区域内的最低可能价值。
```

### 3.2 Bad Information

不适合作为有效竞价信息：

```text
这个箱子感觉很危险。
拍卖师眨了一下眼。
有人说里面有好东西。
这个箱子不太对劲。
```

这些可以作为氛围文本，但底层必须对应结构化事实。

### 3.3 Flavor Wrapping

允许用风味文本包装结构化信息。

示例：

```text
底层事实：箱中至少存在 1 件 Oddity。
表现文本：箱门上贴着一张过期的蓝色异常品封条。
```

```text
底层事实：存在至少 1 件 Contraband。
表现文本：清单上有一行被海关黑笔涂掉了。
```

## 4. Public Intel Design

### 4.1 Purpose

公共信息每轮自动出现，用于：

- 给玩家新的共同锚点。
- 推动价格变化。
- 防止前几轮纯看私查运气。
- 让未使用技能的玩家也能参与推理。

### 4.2 Public Intel Examples

#### Quality Intel

```text
显示随机 2 件物品的品质。
显示箱中最高品质物品的品质。
显示某个区域内最高品质。
```

#### Category Intel

```text
显示最大轮廓物品的分类。
显示是否存在 Oddity。
显示是否存在 Contraband。
显示某个 2x2 物品是否为 Collectible。
```

#### Value Intel

```text
显示公共已知保底价值。
显示一件未知物品的最低可能价值。
显示某个区域内物品总保底。
```

#### Shape / Position Intel

```text
显示最高品质物品所在区域。
显示一个 Oddity 的大致尺寸。
显示最高价值物是否为大件。
```

### 4.3 Public Intel Strength

公共信息不应太强。

过强的问题：

- 所有人估值趋同。
- 竞价变成单纯资金战。
- 私查和技能价值下降。

MVP 建议：

- 公共信息多给品质、分类、保底，不直接给完整名称。
- 完整名称更多来自私人查看或强技能。

## 5. Private Look Design

### 5.1 MVP Private Look

MVP 中，私人查看直接显示完整物品信息：

- 名称。
- 品质。
- 分类。
- 价值。
- 是否可保留。

这样最容易验证信息差是否成立。

### 5.2 Future Tuning

如果私查过强，可以拆成：

- 看名称但不看价值。
- 看品质但不看名称。
- 看分类和最低价值。
- 需要多次查看同一物品才能完整识别。

### 5.3 Why Keep Private Look

即使已有公共信息和技能，也应保留每轮私查。

原因：

- 没技能时也有信息博弈。
- 玩家会产生样本偏差。
- 可以解释玩家突然 Raise 的行为。
- 让其他玩家无法确定你看到了什么。

典型心理：

```text
我连看三件都是垃圾，但他还在加价。
他是看到了唯一的大货，还是疯了？
```

## 6. Active Skill Design

### 6.1 Random Dynamic Skill

每个角色有一组技能，但玩家只能选择“使用技能”，不能选择具体技能。

使用流程：

```text
Player chooses Use Skill
→ System randomly triggers one skill from character pool
→ Player receives private skill result immediately
→ Result is scheduled to become public next round
```

### 6.2 Why Random

随机动态技能解决几个问题：

- 防止玩家每局固定开局使用最强技能。
- 增加即兴判断。
- 让别人无法确定你获得了什么信息。
- 允许玩家用“废信息”诈唬。

### 6.3 One-Round Advantage

技能信息下一轮公开。

这意味着：

- 技能不是永久优势。
- 玩家必须在本轮利用信息。
- 其他玩家知道你用技能后，会怀疑你是否看到了关键内容。
- 如果你不立刻变现，优势会消失。

### 6.4 Skill Timing

推荐：下一轮开始公开。

原因：

- UI 清楚。
- 玩家准确理解“我领先一轮”。
- 方便状态机实现。

备选：下一轮报价后公开。

优点：

- 使用者优势更强。

缺点：

- 节奏更难懂。
- 容易产生争议。

MVP 采用下一轮开始公开。

## 7. Known Floor Value

### 7.1 Definition

已知保底价值是根据当前已知信息计算出的箱子最低可能价值。

分为：

```text
Public Known Floor: 仅基于公共信息。
Your Known Floor: 公共信息 + 你的私有信息。
```

### 7.2 Why It Exists

玩家已经要处理很多事情：

- 轮廓。
- 品质。
- 分类。
- 出价。
- 技能。
- 资金。
- 直接成交。

系统应该帮助玩家整理客观下限，减少算术负担。

### 7.3 What It Should Not Do

系统不应显示：

```text
推荐出价。
推荐 All-in。
预计利润。
箱子期望值精确计算。
```

原因：这会替玩家做决策。

### 7.4 Calculation

每件物品根据当前已知信息筛选候选集合。

```text
itemFloor = min(candidate.value)
containerFloor = sum(itemFloor)
```

例：

```text
已知：物品轮廓为 2x2。
候选：旧收音机 $80，船钟 $450，异常电池 $1600。
保底：$80。
```

若之后知道品质为 Rare：

```text
候选缩小为船钟 $450，异常电池 $1600。
保底：$450。
```

## 8. Bidding Signal Design

报价本身也是信息。

玩家会根据以下行为推测对手：

- 使用技能后是否 Raise。
- 私查后是否突然 Raise。
- 最高价是否快速上升。
- 是否有人 All-in。
- 是否有人 Fold。
- 最高价与第二名差距是否扩大。

因此报价公开策略必须谨慎。

---

## 9. Public Bidding Feedback

### 9.1 Current MVP Feedback

每轮结束公开：

- 当前最高价金额。
- 第一名与第二名差距描述。
- 是否接近直接成交。
- 是否有人 All-in。
- 是否有人 Fold。
- 到期技能信息。

不公开：

- 谁是最高价。
- 每个人具体报价。
- 完整排名。

### 9.2 Why Not Fully Public

完全公开报价会带来：

- 玩家搭便车。
- 报价锚点过强。
- 对物品推理关注下降。
- 若能降价，则前几轮变成无成本表演。

### 9.3 Why Public Highest Amount

MVP 仍公开最高金额。

原因：

- 玩家需要价格锚点。
- 规则更好懂。
- 方便理解直接成交。
- 测试阶段降低认知负担。

高级场可改成最高价区间或模糊资金。

## 10. Commitment Bid Design

### 10.1 Problem It Solves

如果玩家每轮能自由重新报价并降低价格，会出现：

```text
前几轮乱报高价
→ 吓人或阻止成交
→ 后面发现判断错了就降价
→ 错误判断没有代价
```

这会让前 4 轮失去意义。

### 10.2 Current Solution

使用承诺价：

- 只能保持或提高。
- 不能降低。
- 如果判断错，只能 Fold 或继续承担风险。

这让报价成为真实承诺。

### 10.3 Fold Interaction

Fold 不支付承诺价，但代价是退出当前箱子。

这意味着玩家可以诈唬后撤退，但不能最后一轮回来捡漏。

MVP 不额外收 Fold 手续费。

如测试发现高价诈唬仍过多，可加入：

```text
Fold fee = currentCommitmentBid * 5% ～ 10%
```

但不是当前版本。

## 11. Direct Sale Pressure

### 11.1 Purpose

直接成交机制使每轮都有终局可能。

没有直接成交时，玩家会倾向于：

```text
前 4 轮少承担风险
最后一轮信息最多再出合理价
```

直接成交让玩家担心：

```text
如果我不维持竞争价，领先者可能现在就拿走。
```

### 11.2 Thresholds

建议：

```text
Round 1: 200%
Round 2: 160%
Round 3: 130%
Round 4: 110%
Round 5: highest wins
```

### 11.3 Psychological Effect

早期：

- 信息少。
- 阈值高。
- 只有极自信或极想偷鸡的人会压大价。

中期：

- 信息逐渐增加。
- 阈值下降。
- 玩家必须判断是否要防止别人提前成交。

第 4 轮：

- 阈值很低。
- 只要领先明显就可能直接成交。

第 5 轮：

- 没有缓冲，最高价成交。

## 12. All-in as Offensive Action

### 12.1 Why All-in Needs Reward

如果 All-in 仍要满足直接成交阈值，它会变成：

```text
我承担全部风险，但未必获得立即成交奖励。
```

这不够吸引人。

### 12.2 Current Rule

All-in 后若成为最高价，立刻成交。

设计含义：

```text
我放弃后续资金弹性，换取立刻锁定箱子的能力。
```

### 12.3 Risks

All-in 可能导致：

- 过早结束箱子。
- 高波动。
- 新手乱按。

对策：

- UI 明确显示 All-in 后果。
- 可在高级设置中限制第 1 轮 All-in。
- 通过箱子价值范围和托管资金平衡。

## 13. Player Thought Patterns

理想情况下，玩家会产生以下思考：

```text
最高价已经到 $42,000，但差距咬得很紧。
说明至少两个人觉得它值这个价。
我看到的那件 Rare 物品值 $30,000。
公共保底只有 $18,000。
如果还有一个大件是好货，这箱很可能值 $80,000+。
但如果别人是在防直接成交，我可能会接盘。
我要不要 +2 step？
```

或者：

```text
他用了技能后 All-in。
他可能看到了 Oddity。
但也可能抽到废信息在诈。
如果我不跟，他直接拿走。
如果我跟不了，他赢。
```

这些思考比精确算推荐价更重要。

---

## 14. Tuning Levers

可调参数：

- 每轮公共信息强度。
- 私查信息完整度。
- 技能池强度。
- 技能公开时机。
- 已知保底是否显示公共 / 私人两套。
- 直接成交阈值。
- bidStep 大小。
- All-in 是否允许任意轮使用。
- 是否公开最高价身份。
- 是否公开最高价区间而非金额。
